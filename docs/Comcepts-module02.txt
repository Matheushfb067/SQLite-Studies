* Tipos de dados
    INTEGER -> Armazena números inteiros (boolean (0,1), timestamp) 

    TEXT -> Armazena textos (strings, datas, json)

    obs: INTEGER e TEXT são os tipos de dados mais comuns do SQLite!

    REAL -> Números reais/flutuantes (com vírgula)

    BLOB -> Dados binarios (pouco usado!)

    NULL -> Valor nulo (quando iniciamos um campo com valor nenhum)

* Dados no SQLite
    Existem 3 momentos cruciais em que o SQLite lida com tipos de dados:

    Definição de Tabelas -> Definir o tipo de dado da coluna.

    Inserção de Dados -> Converte o dado para o tipo de acordo com a afinidade da coluna.

    Consulta de Dados -> Retorna o valor armazenado.

* Afinidade de Tipo

    SQLite não impõe tipos de dados estritos, mas sim uma afinidade de tipo. Isso significa 
    que você pode armazenar qualquer tipo de dado em qualquer coluna, mas o SQLite tentará 
    converter os dados para o tipo mais apropriado, de acordo com a afinidade da coluna.

    São 5 tipos de afinidade:

    TEXT -> Texto

    INTEGER -> Números inteiros

    REAL -> Números flutuantes

    BLOB -> Dados binários

    NUMERIC -> Números inteiros ou flutuantes   

    Isso significa que por convenção a linguagem, quando digitado por exemplo INT como o tipo 
    de um dado a linguagem automaticamente associa isso ao tipo INTEGER, então mesmo que eu use
    o tipo INT esse atributo será do tipo INTEGER para o SQLite, o mesmo ocorre para os tipos 
    listados acima. Exemplos: usar, INT/SMALLINT/TINYINT para o tipo INTEGER, CHARACTER/VARCHAR 
    para o tipo TEX, DOUBLE/FLOAT para o tipo REAL, DATE/DATETIME/DECIMAL para o tipo Numeric, 
    entre outros...

* Strict 
    No modo strict, o SQLite impõe regras mais rígidas sobre os tipos de dados.

    -> CREATE TABLE -> Só permite tipos de dados especificos, caso contrario retorna um erro na tabela.

    -> INSERT -> quando inserimos o dados, ele terá que ser exatamente do tipo que foi definido no create
    caso contrario retornará um erro.


* Tipos de Dados (STRICT)
    SQLite no modo strict aceita a criação de tabelas com os seguintes tipos de dados:

    INT -> Números inteiros (só se diferencia do integer na criação de chaves primarias -> assunto 
    posterior)

    INTEGER -> Números inteiros

    TEXT -> Texto

    REAL -> Números flutuantes

    BLOB -> Dados binários

    ANY -> Qualquer tipo de dado

* PRIMARY KEY
    Define uma coluna como chave primária, que deve ser única e não nula. É como uma 
    identificação do campo quando o mesmo deve ser especifico somente daquele atributo.

* RowId
    Toda tabela tem uma coluna oculta chamada rowid, que é um identificador único para cada linha.

    Se você não definir uma chave primária, o SQLite cria automaticamente uma coluna rowid.

    Se você definir uma coluna como INTEGER PRIMARY KEY, ela se torna um alias para a coluna rowid, isso 
    não acontee caso use INT.

----------------------------------------------------------------------------------------------------------

* FOREIGN KEY
    Estabelece uma relação entre tabelas, garantindo a integridade referencial.
    
    Imagine que temos as tabelas cursos e aulas, toda aula pertence a algum curso portanto
    dentro da tabela de aulas precisaremos de um atributo que referencie o id da tabela cursos
    por meio do comando: 

    -> FOREIGN KEY("curso_id") REFERENCES "cursos" ("id")

    Enquanto eu posso tem apenas uma chave primaria por tabela, eu posso ter diversas chaves estrangeiras.

* DROP TABLE (FK)
    Para remover uma tabela que possui chaves estrangeiras, você deve primeiro remover as tabelas dependentes.
    Cria uma nova restrição, ou seja não me deixa remover a tabela se em outra tabela tiver uma dependencia de 
    outra por meio da FOREIGN KEY.

    -> DROP TABLE "cursos"; -- Falha se "aulas" existir por conta de dentro da tabela aulas haver a foreign key 
    para a tabela cursos

    De forma resumida, uma tabela depende da outra, sendo assim se quisermos deletar uma, primeiro devemos deletar 
    a outra.

* CASCADE 
    Use o CASCADE para remover/atualizar automaticamente as linhas dependentes ao remover/atualizar uma linha 
    referenciada. De modo mais direto, permite eu deletar os elementos da tabela mesmo com uma foreign key 
    conectando elas.
